## Security findings

### Critical

None found.

### [HIGH] Blind SSRF: `send-web-reminders` POSTs to attacker-controlled `web_push_subscriptions.endpoint` with no allowlist

**File:** `supabase/functions/send-web-reminders/index.ts:158` - **Issue:** The function runs with `SUPABASE_SERVICE_ROLE_KEY`, selects every enabled `web_push_subscriptions` row, and calls `webpush.sendNotification()` against `subscription.endpoint` with no validation against a push-service allowlist. The RLS insert/update policies only enforce `with check (auth.uid() = user_id)` (`supabase/migrations/20260508000000_web_push_notifications.sql:38-51`) and place no constraint on `endpoint`. Any authenticated user can bypass `pushManager.subscribe()` and issue a raw PostgREST upsert with an arbitrary endpoint (e.g. `http://169.254.169.254/latest/meta-data/`, or an internal host:port). The attacker also controls their own `user_preferences` so they can force `reminderKeyIfDue()` to pass and fire on demand. - **Impact:** On the next `*/5` cron tick the service-role worker emits an outbound POST to the attacker's URL from inside project infrastructure. The response body isn't returned (blind SSRF - only 404/410 status disables the row via `classifyPushError`, `_shared/web-reminders.ts:156`), but it enables internal-network probing and request forgery to internal services. - **Recommendation:** Before sending, parse `endpoint` with `new URL()`, require `https:`, and assert the hostname matches a known push-service allowlist (`fcm.googleapis.com`, `*.push.apple.com`, `*.notify.windows.com`, `*.push.services.mozilla.com`) or at minimum a public, non-private/non-loopback/non-link-local IP that is not the metadata host. Enforce this both in the edge function and as a Postgres CHECK/trigger on `web_push_subscriptions.endpoint`. Skip rather than POST to off-allowlist rows.

### [HIGH] Auth tokens left in the web URL/browser history after the callback (implicit flow, no URL scrub)

**File:** `src/lib/supabase.ts:22` - **Issue:** `createClient` is called with no `flowType`, so it defaults to the implicit grant (verified against `@supabase/auth-js`/`GoTrueClient.js:21`). Under implicit flow, Supabase email links (signup confirmation, recovery, magic link) and Google OAuth deliver a long-lived `access_token` **and** `refresh_token` in the URL hash fragment on web. The app sets `detectSessionInUrl: false`, so the SDK's own `history.replaceState`/hash-clearing cleanup never runs, and the manual handler `completeAuthRedirect` reads `window.location.href` (`src/components/app/auth-callback-screen.tsx:27`) but never scrubs the URL afterward. - **Impact:** The refresh token persists in the browser address bar and history. On a shared/public computer it is recoverable from history and grants long-lived account access to a PHI app (mood, journal, thought records, crisis info). - **Recommendation:** Set `flowType: 'pkce'` in the `createClient` auth options so callbacks carry only a single-use short-lived `code` in the query string (the callback already calls `exchangeCodeForSession`). After `completeAuthRedirect` succeeds on web, scrub with `window.history.replaceState(window.history.state, '', '/auth-callback')` (or redirect to a clean path) so no auth material remains in the URL or history.

### [HIGH] Account deletion aborts entirely if client-side avatar cleanup throws, leaving PHI undeleted (GDPR Art. 17)

**File:** `src/features/settings/repository.ts:357` - **Issue:** `deleteUserAccount()` `await`s `removeCurrentUserUploadedAvatar()` **before** invoking the `delete_user_account()` RPC, with no try/catch. `removeCurrentUserUploadedAvatar()` (`src/features/profile/repository.ts:461-476`) issues a `profiles.select()` and on any error throws `getAvatarSetupError(error)`, which maps an RLS error to a thrown `'Profile picture storage permissions are not applied yet...'` (`src/features/profile/repository.ts:94-98`). That throw propagates out before the RPC line is reached. - **Impact:** A transient storage/select error (or an environment where the profile-pics RLS policies were never applied) makes the **entire** account deletion fail - `auth.users` and all PHI rows (`thought_records`, `mood_logs`, `profiles`, etc.) are never deleted, so the right-to-erasure request silently fails for a reason unrelated to data deletion. No test covers this path (`test/integration/db-functions.integration.test.ts:147` calls the RPC directly, never the wrapper). - **Recommendation:** Wrap `removeCurrentUserUploadedAvatar()` in try/catch (or call it only **after** a successful RPC) so storage/permission errors can never block the DB/auth deletion. Better, move authoritative storage deletion server-side (see related finding) and treat the client avatar delete as best-effort.

### [HIGH] GDPR data export omits the entire ACT module, `plan_items`, and `widget_preferences`; no test guards the key set

**File:** `supabase/migrations/20260553_checkin_and_selfcare_dedup.sql:22` (base body) / `supabase/migrations/20260561_export_user_data_restore_modules.sql:13` (latest module wrapper) - **Issue:** `export_user_data()` (the RPC behind Settings → "Export my data", `src/features/settings/repository.ts:365-374`) is maintained as a fragile rename-then-wrap chain. `20260553` re-wrote the whole body in a plain `CREATE OR REPLACE` that did **not** call the prior wrapper, silently truncating everything before it - `20260561`'s own header documents this exact failure (it restored only meditation/gratitude/habits). The live export produces 30 top-level keys and never includes the 11 ACT tables (`act_program_state`, `act_defusion_logs`, `act_expansion_logs`, `act_urge_surf_logs`, `act_connection_logs`, `act_observing_self_sessions`, `act_value_entries`, `act_bulls_eye_snapshots`, `act_committed_actions`, `act_action_steps`, `act_choice_points`), `plan_items` (`20260525_plan_items.sql:3`), or `widget_preferences` (`20260539_widget_preferences.sql:3`) - all user-owned (`user_id ... REFERENCES auth.users`). The integration test asserts an allowlist that stops at `journalEntries`, so adding a table and forgetting the export still passes green; the e2e test only checks the success toast. - **Impact:** Users exercising GDPR Article 15 (access) / Article 20 (portability) receive an export missing all their ACT therapy free-text narratives (`fused_thought`, `defused_version`, `urge_description`, `value_statement`, `barriers`, etc.), their personal action plan, and widget layout - directly contradicting `docs/data-privacy-model.md:20` and `docs/gdpr-compliance.md:19`. Deletion is unaffected (all tables `ON DELETE CASCADE` off `auth.users`); the gap is export-only, which is the legally load-bearing side of a DSAR. - **Recommendation:** Replace the wrapper chain with a single flat `CREATE OR REPLACE` body that builds the whole JSON in one place (drop/REVOKE the `export_user_data_before_*` shadows), adding `WHERE user_id = uid` blocks for all 11 ACT tables, `plan_items`, and `widget_preferences`. Add a regression assertion - a pgTAP/`DO`-block check or an `information_schema`-driven integration test - that every base table with a `user_id` FK to `auth.users` maps to a present export key (with a reviewed allowlist for intentional exclusions like `feedback_submissions`).

### [MEDIUM] React Query cache (sensitive PHI) is never cleared on sign-out

**File:** `src/features/auth/api.ts:162` - **Issue:** `signOut()` only calls `supabase.auth.signOut()`. Nothing calls `queryClient.clear()`/`removeQueries()`, and `onAuthStateChange` (`src/providers/session-provider.tsx:47`) only updates React state. The `QueryClient` is a module-level singleton (`src/providers/app-providers.tsx:22`) holding every fetched PHI record. - **Impact:** After sign-out the previous user's PHI remains resident in the in-memory cache. On web this is partially mitigated by the forced full reload in `ProtectedLayout` (`window.location.href = '/'`, `protected-layout.tsx:53`), but only when sign-out fires from inside `(app)/*` - and **never on native**, where the prior user's health data stays in memory until the process is killed. Cross-user display is limited because query keys embed `userId`; the residual risk is data-at-rest-in-memory after logout. - **Recommendation:** Handle `event === 'SIGNED_OUT'` in the `onAuthStateChange` callback and call `queryClient.clear()` there (fires regardless of which screen triggered sign-out, and on native), or call it in the shared sign-out handler after `supabase.auth.signOut()` resolves.

### [MEDIUM] Signed private-avatar URL (1h) survives sign-out via the un-cleared cache

**File:** `src/features/auth/api.ts:162` - **Issue:** The `profile-pics` bucket is private and avatar URLs are minted as 1-hour signed URLs (`AVATAR_SIGNED_URL_SECONDS = 60*60`, `src/features/profile/repository.ts:6`; `createSignedUrl` at `:233-252`). `mapProfileRow` embeds the signed URL into the `UserProfile` cached under `["profile", userId]`. Because `signOut()` never clears the query client (same root cause as above), the still-valid signed URL remains retrievable from the in-memory cache (e.g. React Query devtools, or before `gcTime` eviction if the screen is revisited). - **Impact:** On a shared/kiosk web browser, after user A signs out, the 1-hour signed URL to A's private avatar stays accessible, exposing personal data across sessions; the signed URL itself remains valid against storage for the full hour regardless of auth state. - **Recommendation:** Purge the query cache on sign-out (handle `SIGNED_OUT` in `session-provider.tsx` and call `queryClient.clear()`, or `removeQueries({ queryKey: ['profile'] })` in the sign-out flow). Additionally consider reducing `AVATAR_SIGNED_URL_SECONDS` to a few minutes since the URL is re-minted on each profile fetch.

### [MEDIUM] `delete_user_account()` RPC performs no storage cleanup, orphaning private avatar objects after erasure

**File:** `supabase/migrations/20260565_emotion_preferences.sql:89` - **Issue:** The live `delete_user_account()` deletes only DB rows then `auth.users`, relying on FK `ON DELETE CASCADE` for the rest. Postgres FK cascade never touches `storage.objects`, and a repo-wide search confirms there is **no** server-side storage cleanup anywhere (no `delete from storage.objects`, no AFTER DELETE trigger on `auth.users` in any of the 60 migrations). The only storage cleanup is the client-side `removeCurrentUserUploadedAvatar()`. - **Impact:** On native/mobile or any path where the client call doesn't run/succeed, the user's uploaded avatar in the private `profile-pics` bucket survives account deletion - orphaned PHI-adjacent personal data (a photo of the user) keyed under the now-deleted user's id folder, which nothing will ever reference or reclaim. GDPR Art. 17 requires it erased. - **Recommendation:** Perform storage deletion server-side inside the SECURITY DEFINER RPC so it cannot be skipped: `delete from storage.objects where bucket_id = 'profile-pics' and (storage.foldername(name))[1] = uid::text` (folder-scoped, covers historical files), or an AFTER DELETE trigger on `auth.users`. Keep the client-side delete only as best-effort.

### [MEDIUM] CSP `script-src` allows `'unsafe-inline'`, defeating a core XSS protection

**File:** `public/_headers:6` - **Issue:** The CSP uses `script-src 'self' 'unsafe-inline'`. With no nonce or hash present, `'unsafe-inline'` is fully active and permits any injected inline `<script>` or event handler to execute - the single most important payload class CSP is meant to block. The allowance appears unnecessary: the production `dist/index.html` loads only one **external** bundle (`<script src="/_expo/static/js/web/entry-...js" defer>`, `dist/index.html:36`) and contains no inline scripts; the only inline content is a `<style id="expo-reset">` block covered by the separate `style-src 'unsafe-inline'`. - **Impact:** An injected inline script could exfiltrate the in-memory Supabase session token or PHI from a sensitive-data app. - **Recommendation:** Tighten to `script-src 'self'`. Keep `'unsafe-inline'` only in `style-src` (react-native-web injects styles dynamically). After changing, run `npm run export:web` and load the build to confirm no inline scripts are injected and the app still functions (the bundle contains `eval`/`new Function` from dependencies; verify `'unsafe-eval'` is still not required).

### [MEDIUM] No per-user cap on `web_push_subscriptions` rows enables SSRF / push-request amplification

**File:** `supabase/migrations/20260508000000_web_push_notifications.sql:6` - **Issue:** There is no limit on how many `web_push_subscriptions` rows a single user may create - the only constraint is `unique(user_id, endpoint)` (`20260560_bug_hunt_rls_security.sql`), which still allows unlimited _distinct_ endpoints. `send-web-reminders` loops over every enabled subscription with no per-user bound (`supabase/functions/send-web-reminders/index.ts:141-214`). - **Impact:** Combined with the unvalidated-endpoint SSRF, one user can insert many rows pointing at arbitrary targets, turning each cron run into many server-originated outbound POSTs (amplified blind SSRF / outbound request flood) and enabling enumeration of many internal targets per run. - **Recommendation:** Cap active subscriptions per user (a trigger rejecting inserts beyond N, or prune oldest), and bound the per-user/total work in one `send-web-reminders` invocation. Severity drops once the endpoint allowlist is in place, but the cap is worthwhile defense-in-depth.

### [LOW] SECURITY DEFINER `export_user_data()` / `delete_user_account()` retain the default PUBLIC EXECUTE grant

**File:** `supabase/migrations/20260565_emotion_preferences.sql:83` - **Issue:** Both account-management SECURITY DEFINER functions are recreated via the rename-then-recreate chain, and PostgreSQL grants EXECUTE to PUBLIC by default on every newly created function. The migrations only `GRANT EXECUTE ... TO authenticated` and never `REVOKE ... FROM public`, so `anon`/`public` retain EXECUTE, reachable via PostgREST at `/rest/v1/rpc/export_user_data` and `/rest/v1/rpc/delete_user_account` with only the anon key. - **Impact:** Not currently exploitable - both functions begin with `if uid is null then raise exception 'Not authenticated'` and `auth.uid()` is NULL for anon, so an unauthenticated call just raises and touches no rows. This is a least-privilege gap, notable because every intermediate `*_before_*` wrapper and `record_feedback_submission()` **are** explicitly revoked from public - these two live functions are the only SECURITY DEFINER RPCs that miss it. - **Recommendation:** In a new migration after `20260565`, add `revoke execute on function public.export_user_data() from public, anon;` and the same for `delete_user_account()`, re-granting to `authenticated`, to match the rest of the schema.

### [LOW] Free-text fields on sensitive-data tables have no server-side length bound

**File:** `supabase/migrations/20260515_cbt_phase3.sql:7` - **Issue:** Most free-text columns storing sensitive health content are plain `text` with no length CHECK (e.g. `core_beliefs.belief_statement`/`reinforcement_plan`, `worry_entries.worry_statement`/`coping_statement`, `thought_records.situation`/`outcome_notes`, `journal_entries.title`/`body`, `anger_logs.trigger_text`, `exposure_*.description`, `mood_logs.notes`). Newer modules added DB caps (`gratitude_entries` `length(note) <= 2000`, `habits` `length(name) <= 120`), but older modules were never backfilled, and some client zod schemas have no `.max()` at all (`src/features/worry/schemas.ts`, `src/features/beliefs/schemas.ts`). Repositories only `.trim()` and insert. - **Impact:** Because supabase-js talks straight to PostgREST, an authenticated user can bypass the UI and POST an arbitrarily large value (tens of MB) into their own row - storage abuse / row-bloat / memory pressure during `export_user_data()` (which `json_agg`s every row). Bounded to the attacker's own data; not a confidentiality bug. - **Recommendation:** Add `check (length(<col>) <= <max>)` constraints to the unbounded sensitive-text columns (mirroring gratitude/habits), with limits consistent with existing client caps (e.g. 20000 for journal body, a few thousand for statements/notes), and add matching `.max()` to the worry and beliefs zod schemas so client and DB agree.

### [LOW] `profiles.display_name` has no server-side length CHECK, breaking the universal free-text bound convention

**File:** `supabase/migrations/20260544_add_display_name.sql:2` - **Issue:** The migration adds `display_name text` with no constraint. The only write path, `updateUserDisplayName` (`src/features/profile/repository.ts:438-459`), upserts `display_name: displayName.trim() || null` with no max-length. The UI bound (`<Input maxLength={100} />`, `settings-screen.tsx:531-538`) is presentational only - it does not constrain the value passed to the mutation (callable directly), and there is no zod schema bounding it. This contradicts the repo's otherwise-universal convention (`habits` `length(name) <= 120`, `gratitude_entries` `length(note) <= 2000`). - **Impact:** A user can persist a multi-megabyte `display_name` to their own row, bloating `profiles`, every `getOrSyncUserProfile` read payload, and any breadcrumb/log that captures the profile. RLS-scoped to `auth.uid() = user_id`, so self-targeted resource abuse, not cross-user. Confirmed: `display_name` only reaches RN `<Text>`/`<Image>` sinks (no `dangerouslySetInnerHTML`/`innerHTML`/`document.title` anywhere), so no XSS; and it is **not** currently in the GDPR export (`export_user_data()` selects only `email, created_at, updated_at` from `profiles`). - **Recommendation:** Add a forward-additive migration: `ALTER TABLE public.profiles ADD CONSTRAINT profiles_display_name_length CHECK (display_name IS NULL OR char_length(display_name) <= 100);` (or 120 to align with `habits_name`), and enforce a max length in `updateUserDisplayName` so direct mutation calls cannot bypass it.

### [LOW] CSP omits `object-src 'none'`

**File:** `public/_headers:6` - **Issue:** The CSP is otherwise thorough (`frame-ancestors 'none'`, `base-uri 'self'`, `form-action 'self'`, `worker-src 'self'`) but does not declare `object-src 'none'`. While `default-src 'self'` covers same-origin `<object>`/`<embed>`, explicitly setting `object-src 'none'` is the recommended hardening - plugin/embedded content is never needed by this SPA and is a known CSP-bypass/clickjacking vector. - **Impact:** Minor hardening gap, not an exploitable flaw. - **Recommendation:** Add `object-src 'none'` to the CSP directive list. Optionally add `report-to`/`report-uri` so CSP violations (including future inline-script regressions) are observable.

### [LOW] Historical/interrupted-upload avatar files are never reclaimed; `removeStoredAvatar` ignores delete failures

**File:** `src/features/profile/repository.ts:461` - **Issue:** Avatar uploads use timestamped, ever-changing paths (`${input.userId}/avatar-${Date.now()}.${extension}`, `:331`). On deletion, `removeCurrentUserUploadedAvatar()` (`:461-476`) deletes only the single current `avatar_storage_path` and only when `avatar_source === 'upload'`. Any prior object from an interrupted upload, an upsert/storage-write race, or a since-reset `avatar_source` is never enumerated or deleted. Compounding this, `removeStoredAvatar()` (`:315-322`) calls `storage.remove([storagePath])` but does not inspect the returned error, so even a current-avatar delete can fail silently. - **Impact:** Orphaned private-bucket avatar objects persist indefinitely after erasure; callers believe cleanup succeeded when it may not have. - **Recommendation:** Delete the entire `{userId}/` folder rather than a single recorded path (naturally idempotent, and folder-scoped if done server-side per the RPC recommendation), and surface/log the result of `storage.remove()` so silent failures are observable.

### [LOW] Erasure is non-atomic across storage and database

**File:** `src/features/settings/repository.ts:355` - **Issue:** `deleteUserAccount()` performs two independent, non-transactional operations - a client-side storage delete (`:357`) followed by the `delete_user_account()` RPC (`:358`) - with no compensating logic. If the storage delete succeeds but the RPC fails (network drop after `:357`), the avatar is gone but the account and all PHI remain; if the storage delete is skipped/swallowed and the RPC succeeds, the account is gone but the avatar persists. - **Impact:** Inconsistent half-deleted state; true atomicity is impossible while storage and SQL are driven as two separate client calls. - **Recommendation:** Consolidate erasure into a single server-side operation (storage delete inside the SECURITY DEFINER RPC, or an AFTER DELETE trigger on `auth.users`) so the whole erasure is one atomic, idempotent unit and any retry converges to "no PHI remains."

## Optimization findings

### [HIGH] `mood_logs` has no index despite per-user time-ordered queries on every dashboard load

**File:** `supabase/migrations/20260514_cbt_phase1.sql:61` - **Issue:** `mood_logs` is created with no index of any kind, yet it is one of the most frequently read tables. `src/features/mood/repository.ts:36-43` runs `.eq('user_id', userId).order('logged_at', {ascending:false}).limit(limit)` - and this exact query runs on the Progress screen (`progress-screen.tsx:58`, limit 200), the mood-trend home widget, and `useCbtInsights` (`use-cbt-insights.ts:82`, limit 60). - **Impact:** With no `(user_id, logged_at)` index, Postgres seq-scans and sorts the entire `mood_logs` table (all users' rows) on each call. Every other later log table got a matching index; `mood_logs` was simply skipped. - **Recommendation:** In a new additive migration, `create index if not exists mood_logs_user_logged_idx on public.mood_logs (user_id, logged_at desc);` (matching the established convention).

### [HIGH] The entire early CBT table family ships with zero indexes

**File:** `supabase/migrations/20260515_cbt_phase3.sql:4` - **Issue:** Migrations `20260514`, `20260515`, `20260516000000`, `20260517`, and `20260525` create `thought_records`, `goals`, `milestones`, `activity_logs`, `values_profile`, `core_beliefs`, `exposure_hierarchies`, `exposure_items`, `exposure_sessions`, `worry_entries`, `procrastination_tasks`, `task_steps`, `anger_logs`, `self_care_logs`, `recovery_plans`, `challenge_plans`, and `plan_items` with **no** indexes. Every `repository.ts` for these features filters by `user_id` and orders by `created_at`/`updated_at` (e.g. `cbt/repository.ts:50-57`, `goals/repository.ts:65-68`, `beliefs/repository.ts:41-44`, `worry/repository.ts:39-42`, `anger/repository.ts:43-46`). - **Impact:** All of these become full table scans + sorts across all users' rows - an internally inconsistent gap, since every table from `20260521` onward got a proper `(user_id, ts)` index. - **Recommendation:** One additive migration with composite indexes per table: `(user_id, updated_at desc)` on `thought_records` (partial `where archived_at is null` to serve `listThoughtRecords`), `(user_id, created_at desc)` on `goals`/`core_beliefs`/`worry_entries`/`anger_logs`/`challenge_plans`/`procrastination_tasks`, `(user_id, scheduled_at)` on `activity_logs`, `(user_id, created_at desc)` on `exposure_hierarchies`, `(user_id, item_order)` on `plan_items`.

### [HIGH] Mood history renders up to 200 logs with `.map()` in a non-virtualized ScrollView, each row spinning up its own query hook

**File:** `src/features/mood/mood-history-list.tsx:31` - **Issue:** `MoodTrackerScreen` passes the full uncapped array (`useMoodLogs(userId, 200)`, `mood-tracker-screen.tsx:45`) to `MoodHistoryList`, which groups via `groupLogsByDate(logs)` on every render and renders every entry through nested `.map()` inside a non-virtualized View/ScrollView - no FlatList/SectionList. Worse, each `MoodEntryCard` (`mood-entry-card.tsx:21`) calls `useEmotionDisplay()` (`use-emotion-display.ts:16`), mounting its own `useEmotionPreferences` observer **and** rebuilding a `Map<string, EmotionPreference>` in `useMemo`. - **Impact:** For a heavy user with 200 logs that is ~200 query observers and ~200 identical Map rebuilds mounted at once, with no row recycling - a long synchronous mount, large memory footprint, and janky scroll. This is the only large list in the app that is not virtualized. - **Recommendation:** Render with a SectionList (or FlashList) so off-screen rows virtualize. Lift emotion resolution out of the row: call `useEmotionDisplay()` once in `MoodHistoryList` and pass `resolveEmotion` down as a prop. Consider capping the visible history (e.g. 50) with a "show more" affordance.

### [HIGH] Quadratic spread-into-Map pattern in `useCbtInsights` over unbounded activity history

**File:** `src/features/cbt/use-cbt-insights.ts:166` - **Issue:** `activityMoodLiftByCategory` groups per-activity mood lifts using `liftsByCategory.set(cat, [...(liftsByCategory.get(cat) ?? []), lift])`, copying the entire accumulated array on every iteration - O(n²) when many activities fall into the small fixed set of `ActivityCategory` values. The source is the user's **full** activity history (`useActivities` → `listActivities` has no limit, `activities/repository.ts:36`). The same anti-pattern appears at `:114` for `moodScoresByDate` (bounded to 60 logs, so smaller blast radius). - **Impact:** A heavy user with hundreds/thousands of completed activities pays `n²/categoryCount` array copies on every render of the CBT home screen. - **Recommendation:** Get-or-create the array once and `arr.push(value)` (mirroring the O(n) bucket pattern already used correctly in `summaries.ts:81-93` and `meditation/insights.ts:54-68`). Apply the same fix at `:114`.

### [HIGH] Five widget-preview images are byte-identical copies of the app icon (~3.1 MB of pure duplication)

**File:** `app.config.ts:127` - **Issue:** `app.config.ts` registers five widget previews (moodcheckin, breathing, grounding, mindfulness, activity), each pointing at a separate 639,646-byte / 1024×1024 PNG. `md5sum` confirms all five are byte-identical to each other **and** to `assets/icon.png`, `adaptive-icon.png`, and `splash-icon.png` (all `415df6c5...`). - **Impact:** The `react-native-android-widget` config plugin embeds these into the APK, so it ships ~3.1 MB (5 × 639 KB) of redundant payload plus ~1.3 MB more of the same image as icon/adaptive-icon/splash. They are also functionally wrong - every widget shows the generic app icon in the picker rather than a representative preview. - **Recommendation:** Generate small, purpose-specific preview PNGs per widget (the picker renders at a few hundred dp, so 250-320 px is plenty), or at minimum point all five `previewImage` entries at a single shared small asset. Deduplicate `icon.png`/`adaptive-icon.png`/`splash-icon.png` if they are intended to be identical artwork.

### [MEDIUM] FK-child queries filter on the FK column with no supporting index

**File:** `supabase/migrations/20260515_cbt_phase3.sql:30` - **Issue:** Child tables are queried by their parent FK but no index exists on those FK columns: `milestones` by `goal_id` (`goals/repository.ts:118-125`), `exposure_items` by `hierarchy_id` (`exposure/repository.ts:126-133`), `exposure_sessions` by `exposure_item_id` (`:177-184`), `task_steps` by `task_id` (`procrastination/repository.ts:120-127`). - **Impact:** Read-path seq-scans, and the parent `on delete cascade` also seq-scans children on every parent delete. - **Recommendation:** Add `(user_id, goal_id)` on `milestones`, `(user_id, hierarchy_id, suds_rating)` on `exposure_items`, `(user_id, exposure_item_id, completed_at desc)` on `exposure_sessions`, `(user_id, task_id)` on `task_steps`.

### [MEDIUM] `useCbtInsights` fans out to 10 queries (several un-indexed, unbounded `select(*)`) and runs on the Home dashboard

**File:** `src/features/cbt/use-cbt-insights.ts:77` - **Issue:** The hook issues ten independent Supabase round-trips per render (`useActivities`, `useAngerLogs`, `useThoughtRecords`, `useAllExposureItems`, `useSelfCareLogs`, `useMoodLogs(60)`, `useSleepLogs(50)`, `useGratitudeEntries(50)`, `useCoreBeliefs`, `useRecoveryPlan`). Seven target un-indexed tables and four are fully unbounded `select('*')`. The hook is consumed by `cbt-distortion-patterns-widget.tsx:14` on the main Home dashboard and by `cbt-home-screen.tsx`. - **Impact:** Opening Home fires a burst of un-indexed seq-scans purely to compute a top-3 distortion count client-side, pulling full rows for ten datasets when the widget only needs the distortion list. - **Recommendation:** For the Home widget, compute just `topDistortions` from a lightweight query, or move the aggregation into a security-definer RPC returning the precomputed payload in one round-trip. At minimum add the missing indexes and cap the unbounded lists.

### [MEDIUM] Lightweight Today widgets mount heavyweight program hooks that fan out to ~11 queries

**File:** `src/features/home/widgets/cbt-programme-widget.tsx:14` - **Issue:** `CbtProgrammeWidget` renders only a small "Phase X of Y" card but calls `useCbtProgram(userId)`, which mounts 11 React Query hooks including `useMoodLogs(userId, 180)` (`use-cbt-program.ts:30-43`). `ActProgrammeWidget` (`act-programme-widget.tsx:14`) does the same via `useActProgram` (~11 hooks, `use-act-program.ts:31-43`). - **Impact:** On a cold cache each present program widget triggers ~11 network round-trips (including a 180-row mood fetch) to populate a card that only needs the program phase. Shared query keys mitigate the cost for later module screens, but the burst is redundant relative to what the card displays. - **Recommendation:** Derive program status from `preferences` alone (it holds `startedAt`/`completedAt`/`phaseIndex`, enough for "Phase X of Y" / not-started / graduated). Split a lightweight `deriveCbtProgramStatus` selector for the widgets, and keep the full multi-query `useCbtProgram`/`useActProgram` only on the dedicated module home screens. This removes ~9 of ~11 fetches per program widget on Today.

### [MEDIUM] `MoodTrackerScreen` recomputes 6+ heavy aggregations over 200 logs on every render with no `useMemo`

**File:** `src/features/mood/mood-tracker-screen.tsx:52` - **Issue:** `getDayMoodSummary`, `getMoodSummary`, `getWeekDelta`, `getDailyAverages`, `getTopEmotions`, and `buildMoodChartData` run directly in the render body (`:52-75`), each iterating/filtering/sorting the up-to-200-element array, none memoized. The component re-renders on every `setTrendDays`, `setChartContainerWidth` (onLayout fires on mount/resize), `setForceOnboarding`, and any `DateBar` change. - **Impact:** Every such re-render re-runs all aggregations (several allocate Dates per element and build Maps) plus `groupLogsByDate` inside `MoodHistoryList`. Only `buildMoodChartData` actually depends on `trendDays`; the week/day summaries recompute needlessly. - **Recommendation:** Wrap each aggregation in `useMemo` keyed on real inputs (`[moodLogs]` for the summaries, `[moodLogs, trendDays]` for the chart). Chart-width layout state should not force the week/day math to recompute.

### [MEDIUM] `react-easy-crop` (DOM/web-only) is statically imported into the native bundle

**File:** `src/components/app/avatar-crop-modal.tsx:3` - **Issue:** `avatar-crop-modal.tsx` does a top-level static import of `Cropper` from `react-easy-crop`, and `settings-screen.tsx:38` statically imports `AvatarCropModal` with no Platform split (no `.web.tsx`/`.native.tsx` variant) and no lazy boundary. `react-easy-crop` is a DOM library - its `react-native` package.json field points at the same web build and its code uses `window.ResizeObserver`. - **Impact:** On iOS/Android this code can never run, yet it is pulled into the native JS bundle via `settings-screen` → `avatar-crop-modal` → `react-easy-crop`, adding dead weight and parse/eval cost. - **Recommendation:** Gate the cropper behind a web-only boundary: split into `avatar-crop-modal.web.tsx` (with the import) and a native stub, or lazy-load via dynamic import only when `Platform.OS === 'web'`. ### [MEDIUM] Onboarding/help illustrations shipped as oversized PNGs (~11 MB) instead of WebP

### [MEDIUM] `useCbtInsights` recomputes ~9 full-history aggregations on every render with no memoization

**File:** `src/features/cbt/use-cbt-insights.ts:88` - **Issue:** The hook derives nine aggregations (`topDistortions`, `exerciseMoodLift`, `activityMoodLiftByCategory`, `beliefReviewSuggestions`, `recurringThoughtSuggestions`, `selfCareTrend`, `angerPattern`, `exposureProgress`) as bare IIFEs, none wrapped in `useMemo`. React Query keeps data identity stable between renders, so memoization collapses the per-interaction recompute to only when underlying data changes.

### [LOW] Unbounded `select(*)` list queries with no `.limit` on per-user-growing tables

**File:** `src/features/cbt/repository.ts:50` - **Issue:** Several list functions fetch every row with `select('*')` and no `.limit()`/`.range()`: `listThoughtRecords` (`cbt/repository.ts:50`), `listGoals` (`goals/repository.ts:62`), `listCoreBeliefs` (`beliefs/repository.ts:38`), `listWorryEntries` (`worry/repository.ts:36`), `listAngerLogs` (`anger/repository.ts:40`), `listActivities` (`activities/repository.ts:36`), `listAllItems`/`listSessions` (`exposure/repository.ts:139,177`), `listTasks` (`procrastination/repository.ts:64`), `listChallengePlans` (`recovery/repository.ts:112`). Widgets often consume the full list just to show one most-recent item or a count (e.g. `cbt-goals-widget.tsx:13` pulls all goals to render one active goal + a count). - **Recommendation:** Add a default `.limit()` (mirroring mood/journal/sleep at 30), and for widgets needing only the latest item or a count, request just that (`.limit(1)` or `head:true` count).

## Notes & limitations

- **Deno edge functions were reasoned from source, not executed.** There is no local Deno runtime in this environment, so `supabase/functions/**` (including the SSRF in `send-web-reminders` and the push-amplification cap) could not be run or fuzzed locally - findings there are derived from static reading of the source and migrations. Likewise, the SQL-side recommendations (indexes, CHECK constraints, RPC revokes) were verified by reading the migrations tree, not by applying them to a live database.
- **Confirmed-but-lower-confidence items** (verified by 2 of 3 reviewers rather than 3 of 3): the avatar-cleanup ordering bug in account deletion, the wrapper-chain GDPR export gap (the higher-confidence framing of the same export issue is 3/3), the React Query cache-not-cleared-on-sign-out finding, the `useCbtInsights` 10-query fan-out, the non-atomic erasure and orphaned-avatar findings, the `I18nProvider` unstable value, and the `expo-system-ui` unused-dependency claim. These remain valid but warrant a confirming check during remediation.
- **Two pairs of findings overlap by design and should be fixed together:** the two GDPR-export findings describe the same root cause at different confidence/scope (the flat-rewrite plus a key-set regression test resolves both), and the two sign-out cache findings (general PHI residue + the specific 1-hour signed avatar URL) are resolved by a single `queryClient.clear()` on `SIGNED_OUT`. The server-side erasure recommendation (storage delete inside `delete_user_account()`) simultaneously addresses the orphaned-avatar, non-atomic-erasure, and avatar-cleanup-abort findings.
- **Explicitly not flagged:** account _deletion_ completeness - every user-owned table FKs `auth.users` with `ON DELETE CASCADE`, so `delete_user_account()`'s `delete from auth.users` does remove all rows (including ACT tables). The GDPR gaps above are export-only; deletion of DB rows is sound. The only deletion gap is storage objects, covered separately.
